# 芯片软件验证自动化平台模块细节

> 配套主图：`architecture/chip-validation-platform-layered.svg`
>
> 本文按五层展开每个模块的职责、输入、输出和关键设计点。后续细化数据库、接口、状态机、Agent 编排时，可以从这里继续拆分。

## 第一层：基础资源与存储层

这一层提供真实资源、数据持久化和版本来源。它不直接表达业务逻辑，但决定平台能否稳定纳管 EMU、FPGA、CPU、QEMU、日志、结果和构建产物。

### SQL DB

- **职责**：保存业务元数据和状态，包括需求、用例、资源、计划、任务、Issue、PR、里程碑、Agent 配置、规则配置。
- **输入**：平台 API 写入、Agent 执行结果、Webhook 事件、人工操作。
- **输出**：管理页面查询、规则引擎判断、Agent 上下文、报表统计。
- **关键设计点**：所有状态变化必须可追溯；核心对象需要状态机字段、owner 字段、版本字段、审计字段；不要把大日志直接塞进 SQL。

### S3 / MinIO

- **职责**：保存大文件和不可结构化产物，包括执行日志、trace、dump、波形、截图、报告、最小复现包。
- **输入**：执行 Agent 上传、日志组件归档、报告 Agent 生成、人工上传附件。
- **输出**：失败分析、问题定位、Issue 复盘、报告下载。
- **关键设计点**：对象路径要绑定项目、计划、任务、资源、版本矩阵；失败现场需要防删除策略；关键 artifact 需要 checksum 和生命周期策略。

### Redis

- **职责**：提供队列缓存、分布式锁、资源心跳、短期状态缓存。
- **输入**：任务队列、资源心跳、Worker 状态、Agent Runtime 临时状态。
- **输出**：调度器获取待执行任务、锁服务判定资源占用、看板展示在线状态。
- **关键设计点**：锁必须有 owner、TTL 和续约机制；资源锁和任务锁分开；队列消息需要幂等键，避免重复执行。

### EMU 资源池

- **职责**：承载中后期 RTL 行为验证、复杂场景验证、trace 定位和跨平台复现。
- **输入**：调度器下发的执行任务、bit/build 版本、软件版本、用例包、调试参数。
- **输出**：执行结果、trace、dump、运行日志、资源健康状态。
- **关键设计点**：资源稀缺，必须支持预约、drain、debug 独占、现场保护；适合高风险、高价值、需要可观测性的任务。

### FPGA 资源池

- **职责**：承载接近真实硬件的软件栈验证、系统级长稳、压力测试和真实环境复现。
- **输入**：bitstream、固件、驱动、runtime、用例、板卡配置。
- **输出**：执行结果、系统日志、串口日志、硬件状态、失败现场。
- **关键设计点**：定位成本高，适合稳定回归和真实硬件依赖任务；失败后优先检查环境，再决定是否迁移到 EMU/QEMU 复现。

### CPU Farm

- **职责**：承载编译、日志分析、QEMU 仿真、离线解析、批量报告生成。
- **输入**：源码、构建任务、日志文件、分析任务、模型仿真任务。
- **输出**：构建产物、分析结果、报告、模型执行结果。
- **关键设计点**：适合横向扩展；需要区分交互式 CPU、批处理 CPU 和高内存分析节点。

### QEMU + cmodel / amodel

- **职责**：承载快速 bring-up、功能回归、模型对比、失败预筛和最小复现。
- **输入**：软件版本、模型版本、用例参数、随机种子、脚本版本。
- **输出**：快速执行结果、模型差异、日志、可复现命令。
- **关键设计点**：cmodel 偏功能正确性，amodel 偏架构/性能行为；适合先筛查再升级到 EMU/FPGA。

### Git Repo / CI 产物

- **职责**：提供 PR、commit、branch、tag、build、binary、image、测试脚本版本来源。
- **输入**：代码提交、PR 事件、CI 构建事件、制品发布事件。
- **输出**：变更影响分析、增量回归触发、修复验证计划、版本矩阵。
- **关键设计点**：每次执行必须绑定完整版本矩阵；PR 事件和构建成功事件要分开处理。

## 第二层：平台组件与能力层

这一层把底层资源包装成可运行、可调度、可观测、可插拔的平台能力，是 Managed Agent 平台和验证业务模块的技术底座。

> 资源管理与调度已开始细化，见 `architecture/resource-management-scheduling.md`。

### K8s / Worker Runtime

- **职责**：运行平台 Worker、Agent Runtime、日志解析任务、报告任务和周期性任务。
- **输入**：执行任务、定时任务、Agent 任务、资源探测任务。
- **输出**：任务执行状态、运行日志、心跳、失败重试事件。
- **关键设计点**：长任务和短任务分队列；Agent 任务需要预算、超时、沙箱和资源限制；支持 Cron、Job、HPA 和节点亲和性。

### Queue / Lock

- **职责**：提供任务队列、优先级、资源互斥、Agent 冲突仲裁和幂等去重。
- **输入**：计划生成的任务、事件驱动触发任务、人工复跑/取消操作。
- **输出**：可执行任务、锁状态、冲突结果、重试事件。
- **关键设计点**：资源锁、任务锁、Issue 锁、Artifact 写锁要分层；同一任务只能有一个 active executor；所有队列消息都要有幂等键。

### Log Pipeline

- **职责**：收集、索引、解析执行日志、系统日志、Agent Trace 和平台运行日志。
- **输入**：执行 Agent 日志、资源系统日志、Agent tool call、Worker 日志。
- **输出**：结构化日志片段、失败摘要、检索索引、告警事件。
- **关键设计点**：日志必须关联 task_id、resource_id、case_id、version_matrix_id；保留原始日志，同时生成结构化索引。

### MCP 组件

- **职责**：管理 MCP Server、Tool Gateway、工具授权、连接健康和调用审计。
- **输入**：MCP Server 注册信息、工具 schema、Agent 调用请求、权限策略。
- **输出**：可用工具列表、工具调用结果、健康状态、审计记录。
- **关键设计点**：工具权限按 Agent、项目、资源类型隔离；危险工具需要审批；工具调用必须记录输入摘要、输出摘要和耗时。

### Skill 组件

- **职责**：管理 Skill 包、参数 schema、版本、依赖、评审、灰度和发布。
- **输入**：Skill 源码或包、说明文档、参数定义、依赖声明、发布审批。
- **输出**：可被 Agent 引用的 Skill、版本清单、执行模板、质量记录。
- **关键设计点**：Skill 要声明适用场景、输入输出、依赖工具和风险等级；发布前需要测试样例和回滚策略。

### Subagent 组件

- **职责**：管理子 Agent 角色模板、上下文隔离、协作拓扑和结果汇总。
- **输入**：主 Agent 分解任务、角色定义、上下文边界、汇总规则。
- **输出**：子任务结果、证据链、冲突意见、汇总结论。
- **关键设计点**：子 Agent 只能访问被授予的上下文和工具；适合日志并行分析、跨平台对比、候选归因投票。

### Artifacts 组件

- **职责**：管理日志、trace、dump、波形、报告、截图、最小复现脚本等产物索引。
- **输入**：S3 对象、执行结果、报告文件、人工附件。
- **输出**：artifact 列表、下载链接、预览信息、失败现场包。
- **关键设计点**：Artifact 索引存在 SQL，大文件存在 S3；关键产物要绑定 hash、保留策略和访问权限。

### Observability

- **职责**：提供平台指标、Agent 指标、资源指标、任务 SLA、成本和质量评分。
- **输入**：任务状态、资源心跳、Agent Trace、队列延迟、工具调用记录。
- **输出**：监控看板、告警、SLA 报告、Agent 质量趋势。
- **关键设计点**：至少跟踪资源利用率、任务等待时间、失败复跑时间、Agent 成本、工具失败率、人工 override 次数。

### Secret / Access

- **职责**：管理凭据、资源访问授权、工具权限、危险动作审批和越权防护。
- **输入**：用户角色、Agent 身份、项目权限、资源权限、Secret 引用。
- **输出**：授权结果、临时 token、审批记录、拒绝原因。
- **关键设计点**：Agent 不直接持有长期 Secret；暂停、抢占、清理现场、释放 FPGA/EMU 等动作需要权限和审计。

## 第三层：管理模块与规则编排层

这一层是平台业务主干。它把底层资源、任务、版本、结果和人工流程变成统一业务对象，并通过规则和事件驱动自动化。

### 需求管理

- **职责**：管理需求拆解、需求状态、需求到用例覆盖、需求变更影响和投片准入项。
- **输入**：需求文档、手工录入、PR 关联、评审意见、Agent 生成建议。
- **输出**：需求列表、覆盖矩阵、变更影响事件、准入完成度。
- **关键设计点**：需求需要关联模块、owner、风险等级、用例集合、里程碑和验收标准。

### 用例管理

- **职责**：管理用例库、用例元数据、平台适配、用例版本、参数、owner 和历史质量。
- **输入**：用例导入、脚本扫描、需求生成、人工维护、历史执行结果。
- **输出**：可调度用例集合、平台适配表、风险权重、增量回归候选集。
- **关键设计点**：用例需要记录适用平台、禁止平台、预计耗时、是否支持断点、是否支持复跑、依赖硬件特性、历史失败类型。

> 用例执行编排已细化，见 `architecture/use-case-execution-orchestration.md`。该模块支持人工标注 QEMU/EMU/FPGA 归属、用例 DAG、基础 gate 阻断，以及 QEMU 预筛失败跳过后续 FPGA/EMU。

### 资源管理

- **职责**：管理 EMU、FPGA、CPU、QEMU 资源的状态、能力、健康、占用、预约和暂停。
- **输入**：资源登记、心跳、健康检查、人工暂停/恢复、Agent 状态回写。
- **输出**：可调度资源池、资源占用视图、健康告警、预约冲突。
- **关键设计点**：资源状态建议包括 idle、running、paused、reserved、unhealthy、upgrading、offline、debug、draining。

### 里程碑管理

- **职责**：管理项目阶段、投片准入、阶段门、完成度、风险和延期影响。
- **输入**：项目计划、准入规则、执行结果、Issue 状态、人工评审结论。
- **输出**：准入状态、里程碑报告、风险 Top、未闭环清单。
- **关键设计点**：里程碑不是静态日期，要绑定需求完成度、用例通过率、阻塞 Issue、资源瓶颈和风险等级。

### 执行计划管理

- **职责**：管理全量回归、冒烟回归、增量回归、夜间回归、专项定位计划。
- **输入**：人工创建、PR/版本事件触发、规则引擎生成、Agent 推荐。
- **输出**：执行任务集合、计划优先级、计划进度、计划风险。
- **关键设计点**：计划需要支持冻结版本矩阵、分批执行、暂停恢复、失败策略和完成条件。

### 执行任务管理

- **职责**：管理具体任务的排队、调度、部署、执行、采集、复跑、迁移、暂停、取消和完成。
- **输入**：执行计划拆分、人工操作、事件触发、Agent 调度结果。
- **输出**：任务状态、执行结果、日志索引、复跑任务、失败分析任务。
- **关键设计点**：任务状态机建议包括 created、queued、scheduled、deploying、running、collecting、passed、failed、rerun_pending、debug_pending、paused、migrating、blocked、cancelled、completed。

### Issue 管理

- **职责**：管理失败归因、缺陷创建、已知问题匹配、owner 分派、修复闭环和风险升级。
- **输入**：失败分析结果、人工创建、外部缺陷系统同步、复跑结果。
- **输出**：Issue 状态、风险等级、owner 任务、修复验证计划、里程碑风险。
- **关键设计点**：Issue 要关联失败任务、用例、资源、版本矩阵、artifact、归因分类和修复 PR。

### PR 管理

- **职责**：管理 PR、commit、CI、代码评审、变更影响、修复验证和准入建议。
- **输入**：Git webhook、CI 结果、人工关联、Agent 影响分析。
- **输出**：影响用例集合、增量回归计划、修复验证结果、准入建议。
- **关键设计点**：PR 不只看 CI pass/fail，还要看影响模块、用例覆盖、相关 Issue 是否闭环。

### 规则引擎

- **职责**：执行准入规则、调度策略、复跑策略、风险策略、告警策略和审批策略。
- **输入**：业务对象状态、版本矩阵、资源状态、历史结果、人工配置。
- **输出**：调度决策、复跑决策、风险升级、人工审批请求、准入结论。
- **关键设计点**：所有自动决策必须保存理由；规则要支持项目级覆盖、灰度和回放验证。

### 事件驱动与流程编排

- **职责**：承载事件总线、状态机、DAG、SLA、补偿流程和跨模块自动化。
- **输入**：需求变更、用例变更、PR 更新、资源心跳、任务完成、Agent 输出。
- **输出**：新任务、复跑流程、定位流程、报告流程、通知流程、补偿动作。
- **关键设计点**：事件要有 schema、幂等键、来源、版本和 trace_id；长流程要支持恢复和人工介入。

> 事件驱动、规则引擎和 Managed Agent 已抽象为通用自动化编排模块，见 `architecture/automation-orchestration.md`。该模块约束“规则命中 → Agent 运行”为 1:1 触发。

## 第四层：Agent 层

这一层放有明确目的的 Agent。Agent 不直接绕过平台操作资源，而是通过管理模块、规则引擎、MCP、Skill 和事件流程工作。

### 资源调度 Agent

- **职责**：根据任务优先级、资源能力、资源健康、版本匹配和时间窗口选择执行资源。
- **输入**：待调度任务、资源状态、资源能力、规则策略、历史执行数据。
- **输出**：资源分配建议、抢占建议、预约建议、drain 建议。
- **关键设计点**：必须解释为什么选择某资源；FPGA/EMU 稀缺资源要优先高价值任务；调度动作必须经过资源锁。

### 用例选择 Agent

- **职责**：根据需求、PR、版本变化、风险等级和历史失败选择回归用例集合。
- **输入**：需求变更、代码变更、用例元数据、历史结果、准入规则。
- **输出**：冒烟集合、增量集合、全量集合、风险加权列表。
- **关键设计点**：要说明为什么选中和为什么不选中；支持邻近模块扩展和高风险升级。

### 执行 Agent

- **职责**：在指定平台上部署版本、拉起环境、执行用例、采集结果、归档日志。
- **输入**：执行任务、资源分配、版本矩阵、用例参数、环境配置。
- **输出**：Result JSON、日志路径、artifact 索引、资源状态回写。
- **关键设计点**：执行入口、结果码、日志目录和 result schema 必须标准化；支持断点续跑和失败现场保护。

### 失败分析 Agent

- **职责**：分析日志、trace、dump、历史失败和版本变化，输出初步归因和下一步建议。
- **输入**：失败任务、日志、artifact、版本矩阵、已知问题库、历史结果。
- **输出**：失败摘要、归因分类、置信度、owner 建议、复跑/定位建议。
- **关键设计点**：目标是减少第一轮人工筛查，不追求完全替代专家；输出必须附证据和置信度。

> 用例失败分析已按 GitHub Issue/PR 思路细化，见 `architecture/failure-analysis-issue-pr.md`。失败进入 Failure Issue，修复和验证进入 Fix/Verification PR，关闭需要 checks 和证据门禁。

### 复跑 Agent

- **职责**：根据失败类型和规则决定同平台、换资源、跨平台或失败子集复跑。
- **输入**：失败分析结果、复跑规则、资源状态、用例参数、历史复跑结果。
- **输出**：复跑任务、复跑策略、稳定失败判定、偶现失败判定。
- **关键设计点**：平台异常和偶现优先复跑；稳定失败转定位；复跑次数和资源消耗要受控。

### 问题定位 Agent

- **职责**：生成定位计划，执行 trace、二分、缩小规模、跨平台对比和最小复现。
- **输入**：稳定失败、artifact、版本矩阵、资源能力、调试规则、专家 runbook。
- **输出**：定位步骤、最小复现脚本、差异分析、需要人工介入的问题。
- **关键设计点**：定位任务与主回归隔离；占用 FPGA/EMU 需要预约或审批；不能破坏失败现场。

### 报告 Agent

- **职责**：汇总执行结果、资源利用率、失败趋势、风险 Top、准入状态并生成报告。
- **输入**：任务结果、资源指标、Issue 状态、里程碑规则、人工备注。
- **输出**：日报、周报、里程碑报告、风险提醒、资源预测。
- **关键设计点**：报告要区分新增失败、持续失败、已恢复失败、阻塞资源和需要人工介入项。

### 变更影响 Agent

- **职责**：分析 PR、commit、版本变化对需求、模块、用例、平台和里程碑的影响。
- **输入**：代码 diff、依赖图、需求映射、用例映射、历史结果。
- **输出**：影响面、增量回归建议、修复验证计划、准入风险。
- **关键设计点**：需要可解释；对低置信影响面要提示人工确认；与 PR 管理和用例选择 Agent 联动。

### 治理守护 Agent

- **职责**：检查 Agent 输出、工具调用风险、权限、预算、审批和质量。
- **输入**：Agent 计划、工具调用请求、权限策略、预算策略、质量规则。
- **输出**：允许、拒绝、需要审批、需要重试或需要人工确认。
- **关键设计点**：关键资源操作、删除 artifact、释放现场、修改高优先级任务等动作必须被治理。

## 第五层：展示与协同层

这一层面向用户，不只是显示数据，还要提供人工决策入口、配置入口和团队协同入口。

### Dashboard

- **职责**：展示资源利用率、执行进度、失败趋势、投片准入、里程碑风险。
- **输入**：指标、任务状态、资源状态、Issue 状态、准入规则。
- **输出**：项目态势、风险 Top、阻塞项、资源瓶颈。
- **关键设计点**：负责人视图关注准入和风险；资源管理员视图关注利用率和健康；Owner 视图关注失败闭环。

### 验证工作台

- **职责**：提供需求、用例、计划、任务、Issue、PR 的日常操作入口。
- **输入**：用户操作、筛选条件、批量操作、人工 override。
- **输出**：对象变更、事件触发、任务提交、审批请求。
- **关键设计点**：要支持项目维度、模块维度、owner 维度和里程碑维度的工作流切换。

### Agent Studio

- **职责**：可视化管理 Agent、MCP、Skill、Subagent、工具权限、运行记录和版本发布。
- **输入**：Agent 配置、Skill 包、MCP Server、Subagent 拓扑、测试样例。
- **输出**：Agent 版本、能力声明、发布记录、运行 trace、质量评估。
- **关键设计点**：Agent 开发和生产运行要隔离；支持灰度、回滚、测试运行和权限审计。

### 报告与通知

- **职责**：输出日报、周报、里程碑报告、夜间执行摘要、风险提醒和阻塞升级。
- **输入**：报告 Agent 输出、指标、Issue、人工备注、通知规则。
- **输出**：邮件、IM、文档、看板卡片、外部系统同步。
- **关键设计点**：报告内容要可追溯到任务、用例、资源和 artifact；风险提醒要能落到 owner 和下一步动作。

### 管理与配置

- **职责**：管理用户、角色、项目空间、字典、规则配置、资源配置、外部集成配置。
- **输入**：管理员配置、项目策略、权限策略、集成密钥引用。
- **输出**：配置版本、权限结果、策略生效事件。
- **关键设计点**：配置需要版本化和审计；策略变更可能影响调度和准入，必须支持预览和回滚。

## 跨层关键协议

### Result JSON

- 每次执行必须输出统一结果结构，包括 task_id、case_id、resource_id、platform、version_matrix、stage、checkpoint、expected、actual、result、reason、artifact_refs。
- 结果需要支持 pass、fail、skip、blocked、timeout、infra_fail 等状态。
- 失败结果必须包含 fail snapshot，至少能定位日志、版本、资源和执行命令。

### Version Matrix

- 每次执行必须绑定芯片 RTL、FPGA bitstream、EMU build、QEMU、cmodel、amodel、firmware、driver、runtime、compiler、test case、脚本、配置文件版本。
- 版本矩阵用于判断失败是否由版本变化引入，也用于增量回归选择。

### Event Schema

- 事件必须包含 event_id、event_type、source、occurred_at、trace_id、idempotency_key、subject_type、subject_id、payload_version、payload。
- 典型事件包括 requirement.changed、case.updated、resource.heartbeat、plan.created、task.finished、issue.updated、pr.updated、agent.completed。

### Resource State

- 建议统一资源状态：idle、running、paused、reserved、unhealthy、upgrading、offline、debug、draining。
- 状态变更必须记录触发者、原因、关联任务、影响范围和是否人工 override。
