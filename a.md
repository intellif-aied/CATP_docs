AI 驱动的软件投片前多平台资源调度与多 Agent 协作需求说明
背景
当前芯片软件投片前验证涉及多个执行平台和资源，包括 FPGA、EMU、QEMU+model、qemu+cmodel、qemu+amodel 等。不同平台在速度、可观测性、问题定位效率、资源稀缺程度、环境稳定性、版本依赖和可验证范围上存在明显差异。
在投片前窗口期，软件验证面临的典型问题包括：
FPGA / EMU 资源稀缺，机器数量有限，且调度主要依赖人工。
人员下班、休息或跨时区协作时，资源经常空闲，验证吞吐没有最大化。
用例执行、失败分析、问题复现、版本切换、资源暂停、定位实验之间相互影响。
多个平台执行结果缺少统一调度、统一状态、统一问题归因和统一复跑策略。
当某个资源被暂停用于定位问题时，其他验证任务缺少自动迁移和调度能力。
用例更新、版本更新、环境更新后，缺少自动增量识别和影响范围评估。
多个 Agent 或自动化脚本之间容易抢资源、重复执行、状态不一致或互相覆盖结果。
因此，需要算法团队提供一套面向芯片软件投片前验证的 AI 资源调度与多 Agent 协作能力，目标是在有限资源下提升验证吞吐、减少人工盯守、降低平台空闲率，并确保问题定位、用例更新、版本切换不会影响其他验证任务的持续推进。

建设目标
2.1 核心目标
构建一个 AI 驱动的软件投片前验证资源调度与多 Agent 协作系统，使 FPGA、EMU、QEMU+model 等平台可以在统一调度策略下持续运行，实现：
人下班，资源不下班
夜间、周末、午休等无人值守时间，系统仍能自动分配任务、执行用例、收集结果、复跑失败、归档日志。
资源一旦空闲，AI 调度系统能够自动选择合适任务填充执行。
避免 FPGA / EMU 等稀缺资源因人工排队、审批、无人值守导致长时间闲置。
多平台统一调度
支持 FPGA、EMU、QEMU+model、qemu+cmodel、qemu+amodel 等平台的统一纳管。
根据用例类型、平台能力、资源状态、版本匹配关系、优先级和风险等级自动选择执行平台。
支持同一用例在不同平台上的分层执行、对比执行和回归执行。
多 Agent 协作
支持多个 Agent 分工协作，包括调度 Agent、执行 Agent、日志分析 Agent、失败归因 Agent、复跑 Agent、问题定位 Agent、报告 Agent 等。
Agent 之间通过统一任务状态、资源状态、结果数据库和事件队列协同，避免重复执行和互相覆盖。
支持任务暂停、恢复、迁移、抢占和隔离。
资源暂停不影响全局执行
当某个 FPGA / EMU / QEMU 资源需要暂停用于问题定位、环境升级、版本切换或人工调试时，系统应自动冻结该资源上的任务状态。
其他资源继续运行，不应因为单个资源暂停导致整体回归停摆。
被暂停任务可按策略恢复、迁移或等待。
问题定位与批量回归解耦
问题定位任务应与大规模回归任务隔离。
某个失败 case 进入定位模式后，不应阻塞其他 case 的继续执行。
支持将失败用例自动转入更适合定位的平台，例如从 FPGA/EMU 失败后迁移到 QEMU+model 复现，或从 QEMU+model 迁移到 EMU/FPGA 验证真实硬件行为。
用例增量更新不影响其他用例
当某部分用例被修改、禁用、新增或调整参数时，系统应识别影响范围。
只对受影响的用例、模块、平台、版本组合进行增量重跑。
未受影响的用例继续执行，不应因局部更新导致全量任务重启。

角色与 Agent 分工
建议系统至少包含以下 Agent。Agent 不一定对应独立进程，也可以是系统中的逻辑模块。
4.1 资源调度 Agent
职责：
维护所有平台资源状态。
根据任务优先级、资源可用性、平台能力和版本匹配关系进行调度。
支持任务排队、抢占、暂停、恢复、迁移。
避免多个任务抢占同一资源。
避免低优先级长任务占用关键资源。
关键能力：
资源空闲自动填充任务。
夜间自动选择适合长时间运行的任务。
FPGA / EMU 资源优先分配给高价值、高风险、真实硬件依赖强的任务。
QEMU+model 资源优先承担快速复现、日志补充、基础回归和失败预筛。
4.2 用例选择 Agent
职责：
根据投片风险、模块优先级、历史失败率、代码变更、平台能力选择用例。
支持全量回归、增量回归、冒烟回归、风险回归。
避免无意义重复执行。
关键能力：
根据代码变更自动选择影响用例。
根据版本差异自动生成回归集合。
根据历史失败和高风险模块提高调度权重。
对无法在某平台执行的用例自动过滤或降级。
4.3 执行 Agent
职责：
在指定平台拉起环境、部署版本、执行用例、采集日志。
处理执行超时、资源掉线、脚本异常和平台异常。
将执行结果写入统一结果数据库。
关键能力：
标准化执行入口。
标准化日志目录。
标准化结果返回码。
支持断点续跑。
支持失败自动归档。
4.4 失败分析 Agent
职责：
对失败 case 的日志、trace、dump、平台状态、版本信息进行分析。
初步判断失败类型和归因方向。
生成问题摘要，减少人工筛查成本。
失败分类建议：
类型
说明
用例问题
case 本身错误、参数错误、预期错误
软件问题
驱动、runtime、firmware、编译器、库问题
平台问题
FPGA/EMU/QEMU 环境异常、资源掉线、脚本失败
设计问题
RTL / IP / SoC 设计行为异常
模型问题
cmodel/amodel 与预期或硬件不一致
版本问题
硬件版本、软件版本、用例版本不匹配
非确定性问题
偶现、时序相关、资源竞争、初始化顺序相关

4.5 复跑 Agent
职责：
根据失败类型和失败置信度决定是否自动复跑。
对偶发失败、平台异常、环境异常优先复跑。
对稳定失败转入问题定位流程。
关键能力：
支持同平台复跑。
支持换资源复跑。
支持跨平台复跑。
支持缩小参数复跑。
支持只复跑失败子集。
4.6 问题定位 Agent
职责：
对失败 case 自动生成定位计划。
根据问题类型选择合适平台和定位手段。
在不影响主回归的情况下占用指定资源进行调试。
典型定位动作：
在 QEMU+model 上快速复现。
在 EMU 上打开更多 trace。
在 FPGA 上保留现场并暂停资源。
缩小用例规模。
二分版本。
二分参数。
对比 cmodel / amodel / RTL / FPGA 行为差异。
自动提取最小复现用例。
4.7 报告 Agent
职责：
汇总多平台执行结果。
生成日常进展、风险报告、资源利用率报告和失败趋势报告。
输出给项目、软件验证负责人、平台负责人和领域 owner。
报告内容：
当日执行数量。
通过率 / 失败率 / 阻塞率。
FPGA / EMU / QEMU 资源利用率。
Top 失败模块。
Top 阻塞资源。
Top 风险领域。
新增失败、已恢复失败、持续失败。
需要人工介入的问题列表。

核心业务流程
5.1 夜间无人值守执行流程
flowchart TD
    A[人员下班前提交任务池] --> B[AI 调度 Agent 检查资源状态]
    B --> C{资源是否空闲}
    C -- 是 --> D[选择高优先级任务]
    C -- 否 --> E[进入等待队列]
    D --> F[执行 Agent 部署版本并运行用例]
    F --> G[采集日志和结果]
    G --> H{是否失败}
    H -- 否 --> I[记录通过并调度下一个任务]
    H -- 是 --> J[失败分析 Agent 初步归因]
    J --> K{是否适合自动复跑}
    K -- 是 --> L[复跑 Agent 执行复跑]
    K -- 否 --> M[进入问题定位队列]
    L --> I
    M --> N[报告 Agent 输出风险]

5.2 资源暂停与任务迁移流程
flowchart TD
    A[资源需要暂停] --> B[冻结当前任务状态]
    B --> C[保存日志/版本/环境/执行进度]
    C --> D{任务是否可迁移}
    D -- 可迁移 --> E[调度到其他资源继续执行]
    D -- 不可迁移 --> F[保持挂起等待恢复]
    E --> G[其他资源继续运行]
    F --> G
    G --> H[全局回归不受单资源暂停影响]

5.3 用例增量更新流程
flowchart TD
    A[用例/参数/版本发生更新] --> B[识别变更范围]
    B --> C[计算影响模块和平台]
    C --> D[生成增量回归集合]
    D --> E[只重跑受影响用例]
    E --> F[未受影响用例继续执行]
    F --> G[合并新旧结果生成报告]

6.3 调度策略要求
系统应支持以下调度策略：
优先级调度
投片高风险领域优先。
阻塞投片准入的问题优先。
FPGA / EMU 稀缺资源优先给必须依赖真实硬件或 RTL 行为的任务。
资源亲和性调度
需要真实硬件行为的任务优先 FPGA。
需要较好可观测性的任务优先 EMU。
需要快速复现和基础筛查的任务优先 QEMU+model。
长时间低风险回归可放到夜间 QEMU / FPGA 空闲资源。
时间窗口调度
白天优先执行需要人工及时分析的任务。
夜间优先执行长耗时、稳定、自动化程度高的任务。
周末优先执行全量回归、压力测试、长稳测试。
失败驱动调度
新失败优先复跑。
高置信稳定失败进入定位。
偶现失败增加重复执行次数。
同类失败聚类后只保留代表 case 深入定位。
资源保护调度
FPGA / EMU 不应被低价值任务长时间占用。
对长任务设置 checkpoint 或阶段性退出点。
支持资源预约和抢占。
支持资源 drain 模式，避免强制中断导致现场丢失。

多 Agent 协作需求
7.1 协作原则
所有 Agent 必须基于统一任务 ID 工作。
所有状态变化必须写入统一状态中心。
Agent 之间不得直接覆盖彼此结果。
每个 Agent 的动作需要可追溯。
所有自动决策需要保留理由，便于人工审核。
支持人工 override，但 override 需要记录。

问题定位需求
9.1 失败后自动处理策略
失败后，系统应按以下优先级处理：
检查是否为平台异常或资源异常。
检查是否为已知问题。
检查是否为版本不匹配。
检查是否为用例自身问题。
判断是否需要同平台复跑。
判断是否需要跨平台复跑。
判断是否需要进入 debug 资源池。
生成问题摘要和推荐 owner。
9.2 跨平台定位策略
失败平台
推荐动作
QEMU+cmodel 失败
优先判断软件 / 用例 / model 问题
QEMU+amodel 失败
对比 cmodel 结果，判断功能还是性能模型问题
EMU 失败
打开 trace，缩小 case，必要时对比 QEMU+model
FPGA 失败
先检查环境稳定性，再迁移到 EMU/QEMU 尝试复现
多平台均失败
高概率为软件栈 / 用例 / 设计共性问题
仅真实硬件平台失败
高度关注 RTL / 时序 / 环境 / 初始化 / 并发问题

非功能需求
13.1 稳定性
调度系统自身不可成为单点故障。
Agent 异常退出后可恢复。
执行任务异常中断后可追溯。
资源状态异常需要自动降级或隔离。
13.2 可追溯性
每一次调度、暂停、复跑、迁移、失败归因都需要记录：
谁触发。
何时触发。
为什么触发。
输入是什么。
输出是什么。
影响了哪些任务。
是否人工 override。
13.3 可解释性
AI 调度不能只给结果，必须给出决策理由。例如：
为什么该用例被分配到 FPGA。
为什么失败 case 没有立即复跑。
为什么该任务被抢占。
为什么某些用例被纳入增量回归。
为什么某资源被判定为 unhealthy。
13.4 隔离性
问题定位任务与批量回归任务隔离。
不同芯片版本任务隔离。
不同软件版本任务隔离。
不同平台环境任务隔离。
失败现场保护与新任务调度隔离。
13.5 安全性
重要资源暂停、释放、抢占需要权限控制。
自动清理日志或环境需要保护机制。
不允许误删失败现场。
不允许覆盖关键日志。
不允许未授权修改任务优先级或 owner。

验收标准
14.1 第一阶段：基础纳管
支持 FPGA / EMU / QEMU+model 资源统一登记。
支持资源状态展示。
支持任务提交、排队、执行和结果归档。
支持人工暂停、恢复资源。
支持基础日报输出。
14.2 第二阶段：自动调度
支持资源空闲自动分配任务。
支持基于优先级、平台能力、预计耗时的调度。
支持夜间无人值守执行。
支持失败自动复跑。
支持资源 unhealthy 自动隔离。
14.3 第三阶段：多 Agent 协作
支持调度、执行、失败分析、复跑、报告等 Agent 协同。
支持任务暂停、迁移、恢复。
支持问题定位任务与主回归隔离。
支持 Agent 冲突仲裁。
14.4 第四阶段：智能增量与风险收敛
支持用例变更影响分析。
支持版本变更影响分析。
支持增量回归自动生成。
支持失败聚类和初步归因。
支持投片风险 Top 自动输出。
支持资源利用率和里程碑风险预测。

需要算法团队重点解决的问题
如何建模任务优先级
不只是人工优先级，还要结合投片风险、历史失败率、资源稀缺程度、用例覆盖价值、版本变更影响。
如何建模资源价值
FPGA / EMU 资源比 QEMU 资源更稀缺，需要避免低价值任务占用。
资源稳定性、历史失败率、执行速度也应参与调度决策。
如何做多 Agent 协同
调度、执行、复跑、定位、报告之间必须有统一状态机。
需要解决 Agent 间冲突、重复动作、状态覆盖问题。
如何做失败自动归因
需要结合日志、历史失败、版本变化、平台差异、已知问题库进行初步判断。
目标不是完全替代人工，而是减少人工第一轮筛查成本。
如何做增量回归选择
用例变更、软件版本变更、模型版本变更、平台版本变更都可能影响回归范围。
系统需要给出为什么重跑、为什么不重跑的解释。
如何保障不中断执行
某个资源暂停、某个用例更新、某个平台异常，都不能让整体回归停止。
系统需要自动迁移、隔离、恢复和继续调度。

总结
本需求的核心不是简单做一个任务队列，也不是单纯做自动化脚本，而是要面向芯片软件投片前验证建立一个 AI 驱动的多平台资源调度与多 Agent 协作系统。
它需要解决的本质问题是：
人不在时，验证资源仍然持续工作。
多个平台统一调度，避免 FPGA / EMU 等稀缺资源浪费。
多个 Agent 协作，避免人工成为调度和分析瓶颈。
问题定位、用例更新、资源暂停都不影响其他任务继续推进。
投片前风险、进度、资源利用率和失败收敛情况可以被实时看清楚。
最终目标是提升投片前软件验证吞吐、缩短问题发现和定位时间、降低资源空闲率，并让项目团队对投片准入状态和剩余风险有清晰、可信、可追溯的判断。
